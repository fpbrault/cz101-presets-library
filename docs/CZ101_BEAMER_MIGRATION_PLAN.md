# Migration Plan: clack в†’ beamer for CZ-101 Synth VST/AU Plugin

## Overview

Replace the `cosmo-pd101` crate's clack/CLAP layer with [beamer](https://github.com/helpermedia/beamer) to gain native AUv2 + AUv3 + VST3 support via a unified build system. The `cosmo-synth-engine` DSP engine and React WebView GUI remain unchanged.

**Current stack:** clack (CLAP) в†’ clap-wrapper (AUv2, VST3) вЂ” AUv3 is empty stubs
**Target stack:** beamer (AUv2, AUv3, VST3) вЂ” single codebase, one xtask bundler

## Execution Status (2026-04-15)

- Phase 1 is implemented in `cosmo-pd101`: beamer `#[derive(Parameters)]` is live.
- Phase 2 processor wiring has a critical AU host-control fix:
    - `CzProcessor::process()` now applies the full beamer parameter snapshot to DSP every block.
    - `CzParameters::to_synth_params()` now maps global, line, FX, LFO, filter, and portamento controls instead of only a small global subset.
- Impact: AU default (host-native) controls and automation now propagate to the synth engine, not just the custom web UI path.
- Bundle metadata is now project-branded instead of hardcoded beamer defaults:
    - `bundle_identifier_prefix` is read from `Config.toml` and applied to AUv2, AUv3, and VST3 `CFBundleIdentifier` values.
- Build workflow now supports an AUv3-inclusive path without changing the stable default:
    - default scripts still build/install AUv2 + VST3.
    - full-format scripts are available via `build:plugin:all` and `plugin:build-install:all`.

## Why beamer

- Native AUv2 + AUv3 support via shared C-ABI bridge with ObjC wrapper
- AUv3 is required for iOS support (AUv3 is the only Audio Unit format for iOS)
- Built-in webview demo with React + Vite + Tailwind (matches existing GUI setup)
- Declarative parameter system via `#[derive(Parameters)]` macro
- Unified xtask bundler for all formats (`.component`, `.app/.appex`, `.vst3`)

## Migration Decisions (Confirmed)

| Decision | Choice |
|----------|--------|
| Target formats | AUv2 + AUv3 + VST3 |
| Migration strategy | Full replacement (keep clack version in a git branch for rollback) |
| Envelope handling | WebView IPC only via `WebViewHandler::on_invoke()` (not CLAP params) |
| Webview assets | Vite outputs directly to `cosmo-pd101/webview/dist/` |

## Phase 1: Parameter Migration

This is the largest mechanical change. 73 scalar parameters move from manual numeric-ID constants to beamer's declarative `#[derive(Parameters)]` system.

### 1.1 Parameter Type Mapping

| Current Pattern | Beamer Pattern |
|---|---|
| `const P_VOLUME: u32 = 0` | `#[parameter(id = "volume", name = "Volume", default = 0.5, range = 0.0..=1.0)] pub volume: FloatParameter` |
| Enum params with manual `*_to_f64()` / `f64_to_*()` | `EnumParameter<WarpAlgo>` with `#[derive(EnumParameter)]` |
| Parameter info via `get_info()` / `value_to_text()` | Auto-generated by `#[derive(Parameters)]` |
| Bypass parameter | `#[parameter(bypass)] pub bypass: BoolParameter` |

Replace the 4-struct clack pattern with beamer's 2-struct typestate pattern.

### 2.1 Before (clack)

### 2.3 CzDescriptor Definition

```rust
#[beamer::export]
#[derive(Default, HasParameters)]
pub struct CzDescriptor {
    #[parameters]
    pub parameters: CzParameters,

    // Envelopes live here as IPC-only state (not CLAP params)
    pub envelopes: Arc<Mutex<HashMap<EnvId, StepEnvData>>>,

    // Shared IPC for oscilloscope
    pub scope_sender: Sender<ScopeData>,
}

impl Descriptor for CzDescriptor {
    type Setup = (SampleRate, MainInputChannels);
    type Processor = CzProcessor;

    fn prepare(self, (sample_rate, channels): (SampleRate, MainInputChannels)) -> CzProcessor {
        let dsp = Cz101Processor::new(sample_rate.0 as f32);
        CzProcessor {
            parameters: self.parameters,
            envelopes: self.envelopes,
            scope_sender: self.scope_sender,
            dsp,
            sample_rate: sample_rate.0 as f32,
        }
    }

    fn wants_midi(&self) -> bool { true }

    fn webview_handler(&self) -> Option<Arc<dyn WebViewHandler>> {
        Some(Arc::new(CzWebViewHandler {
            envelopes: self.envelopes.clone(),
            scope_sender: self.scope_sender.clone(),
        }))
    }

    fn input_bus_info(&self, index: usize) -> Option<BusInfo> {
        match index {
            0 => Some(BusInfo::stereo("MIDI Input")),
            _ => None,
        }
    }

    fn output_bus_info(&self, index: usize) -> Option<BusInfo> {
        match index {
            0 => Some(BusInfo::stereo("Output")),
            _ => None,
        }
    }
}
```

### 2.4 CzProcessor Definition

```rust
#[derive(HasParameters)]
pub struct CzProcessor {
    #[parameters]
    pub parameters: CzParameters,

    pub envelopes: Arc<Mutex<HashMap<EnvId, StepEnvData>>>,
    pub scope_sender: Sender<ScopeData>,

    dsp: Cz101Processor,
    sample_rate: f32,
}

impl Processor for CzProcessor {
    fn process(&mut self, buffer: &mut Buffer, _aux: &mut AuxiliaryBuffers, _context: &ProcessContext) {
        // 1. Read parameter changes (auto-handled by HasParameters)
        let params = self.parameters.current();

        // 2. Apply params to DSP (batched via params.apply())
        self.dsp.set_params(params.clone());

        // 3. Apply envelope changes from IPC
        if let Ok(envs) = self.envelopes.lock() {
            for (id, env_data) in envs.iter() {
                self.dsp.set_envelope(*id, env_data);
            }
        }

        // 4. Render
        let mut buf = buffer.as_slice();
        self.dsp.process(&mut buf);

        // 5. Scope data
        let scope_data = self.dsp.get_scope_data();
        let _ = self.scope_sender.try_send(scope_data);
    }

    fn wants_midi(&self) -> bool { true }

    fn process_midi(&mut self, input: &[MidiEvent], output: &mut MidiBuffer) {
        for event in input {
            match event {
                MidiEvent::NoteOn { note, velocity, channel, .. } => {
                    let freq = 440.0 * 2.0f32.powf((note as f32 - 69.0) / 12.0);
                    self.dsp.note_on(*note, freq, *velocity as f32 / 127.0);
                }
                MidiEvent::NoteOff { note, channel, .. } => {
                    self.dsp.note_off(*note);
                }
                MidiEvent::CC { controller: 64, value, .. } => {
                    self.dsp.set_sustain(*value >= 64);
                }
                MidiEvent::PitchBend { value, .. } => {
                    self.dsp.set_pitch_bend(*value as f32 / 8192.0 - 1.0);
                }
                _ => {}
            }
        }
    }
}
```

### 2.5 MIDI Event Mapping

| Current (raw bytes) | Beamer MidiEvent |
|---|---|
| `0x90 \| ch, key, vel` | `MidiEvent::NoteOn { channel, note, velocity }` |
| `0x80 \| ch, key, vel` | `MidiEvent::NoteOff { channel, note, velocity }` |
| `0xE0 \| ch, lsb, msb` | `MidiEvent::PitchBend { channel, value }` |
| `0xB0 \| ch, 64, val` | `MidiEvent::CC { channel, controller: 64, value }` |
| `0xB0 \| ch, 1, val` | `MidiEvent::CC { channel, controller: 1, value }` (mod wheel) |

### 3.2 IPC Bridge (JS Side)

Replace the current wry IPC mechanism with beamer's `__BEAMER__` API.

**Old (wry):**
```javascript
window.__czIpc.postMessage(JSON.stringify({ parameter_id: 0, value: 0.5 }));
window.__czIpc.postMessage(JSON.stringify({ envelope_id: 'l1_dco', data: stepEnvData }));
```

**New (beamer):**
```javascript
// Scalar params
await __BEAMER__.invoke('setParam', [0, 0.5]);

// Envelopes (JSON serialized)
await __BEAMER__.invoke('setEnvelope', ['l1_dco', stepEnvData]);

// Oscilloscope (Rust в†’ JS via on_event)
window.__BEAMER__._onEvent('scope', (data) => {
  // data.samples, data.sampleRate, data.frequency
  updateScope(data);
});
```

A small JS shim layer (`src/lib/ipc.ts`) should be created that wraps `__BEAMER__.invoke()` and provides the same interface the current GUI expects. This minimizes changes to the existing React components.

                let data: StepEnvData = serde_json::from_value(args[1].clone()).unwrap();

            }

    }

### 3.5 Standalone AUv3 UI Host App

Goal: provide a real standalone app that opens a window and can play the plugin without a DAW.

Important distinction:
- Current AUv3 container app is registration-only (`LSBackgroundOnly`), so launching it shows no UI.
- This phase adds a foreground macOS host app that instantiates the AUv3 extension and embeds its view controller.

Implementation outline:

1) App behavior and lifecycle
- Change AUv3 container app plist generation to foreground mode for standalone builds (`LSBackgroundOnly = false`).
- Add `NSApplicationSceneManifest` / normal app lifecycle so the app opens a main window.
- Keep existing registration behavior for non-standalone packaging path if needed (build flag split).

2) Standalone host runtime (Swift/ObjC shim)
- Add a tiny host binary in the app target that:
    - Creates an `AVAudioEngine`.
    - Locates the plugin `AudioComponentDescription` (type `aumu`, subtype `CZpd`, manufacturer `FpbC`).
    - Instantiates `AUAudioUnit` via `AVAudioUnit.instantiate`.
    - Connects plugin node to output (`engine.outputNode`) and starts engine.
- Add MIDI input path:
    - Listen to CoreMIDI sources.
    - Forward MIDI to AU event API so external keyboard works.
    - Optionally add a computer-keyboard fallback for smoke tests.

3) UI embedding
- Request plugin view controller from AU (`requestViewController`).
- Embed in host app window (AppKit `NSViewController` container).
- Window size policy:
    - Default 1280x800 from Config.
    - Respect plugin-provided preferred size when available.

4) Preset and state wiring
- Ensure host can trigger AU state save/load (factory presets + user patch recall).
- Keep parameter automation path host-native (same AU params as DAW).

5) Build integration
- Add `--standalone` (or `--auv3-standalone`) mode to xtask AUv3 bundler.
- Produce two AUv3 app variants if needed:
    - Registration container (background-only).
    - Standalone host app (foreground).
- Default local dev command should build/install standalone variant for quick manual testing.

6) Validation checklist
- Double-clicking `CzSynthVst.app` opens a visible window.
- On-screen UI is interactive and audible without DAW.
- External MIDI keyboard triggers notes.
- AUv3 still appears in Logic/GarageBand instrument menus.
- No AUv2/AUv3 duplicate ObjC class collisions at runtime.

## Risk Areas and Mitigations

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Beamer is a newer/less-documented framework | Medium | Pin to specific git rev; keep clack backup branch |
| Oscilloscope 30fps through WebViewHandle | Medium | Benchmark after migration; beamer's `emit()` is async |
| `EnumParameter` with 14 variants (WarpAlgo) | Low | beamer supports arbitrary enum variant count |
| Complex shared state (envelopes via Mutex) | Low | Same pattern as current; works in clack |
| VST3/AUv2 AUv3 bundle generation via xtask | Medium | Study beamer's xtask code thoroughly before adapting |
| WebView IPC throughput (73 params + envelopes) | Low | Current wry approach handles this; beamer similar |
| UniFFI iOS path through beamer | Unknown | beamer AUv3 targets macOS first; iOS may need separate work |
